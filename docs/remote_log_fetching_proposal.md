# Remote Log Fetching Proposal

## Overview
The goal is to allow the Hub's web interface to display the internal `IceHubLog` entries of remote nodes (Nanos). This requires sending a request over the radio, allowing the remote node to package its logs, and transmitting them back to the Hub via a Multipart message.

## Architectural Challenge: Sync vs. Async
REST APIs (`GET /api/nodes/{id}/logs`) are **synchronous**. The browser expects an immediate response.
Radio communication is **asynchronous** and lossy.

If the Hub blocks the HTTP response while waiting for the radio to fetch the logs, the entire ESP32 WebServer will hang. If the remote node is offline or a packet drops, the HTTP request times out gracefully, but the ESP32 might be blocked from handling other important tasks.

### The Refined Solution: Active Streaming with a "Lease"
Instead of repeatedly polling and transmitting the entire log buffer over the radio (which consumes heavy bandwidth), the Hub will instruct a specific Remote Node to *actively stream* its new logs as they happen.

To prevent the remote node from streaming forever if the user closes their browser, we will use a **Leased Subscription** model:
1. **Intent:** The web UI polls `GET /api/nodes/{id}/logs`. 
2. **Lease Renewal:** If `{id} > 0`, the Hub records that `{id}` is actively being monitored. If it hasn't sent a "Stream Request" to that node in the last 10 seconds, it sends one.
3. **Streaming:** The Remote Node receives the request and sets a `log_stream_timeout` to 15 seconds. If this is a *new* lease (transitioning from inactive to active), the Remote instantly transmits its currently cached log history. From that point on, any *new* log entries generated by `IceHubLog` are streamed as they happen.
4. **Cache:** The Hub receives these live log packets, appends them to a small cache for that node, and immediately serves them on the next UI poll.
5. **Graceful Decay:** If the user closes the web UI, the Hub stops renewing the lease. The Remote Node's 15-second timer expires, and it silently stops transmitting logs over the radio.

## Protocol Enhancements

### 1. Requesting Log Stream (Hub -> Remote)
We will reuse the existing `PACKET_CONFIG`.
New Addition: `command = 2` means "Enable Log Streaming".

### 2. Transmitting Live Logs (Remote -> Hub)
Since standard logs are up to 64 bytes, they slightly exceed the standard 29-byte payload limit. We will use `PACKET_MULTIPART`.
New Addition: Multipart `header.type = 2` means "Log Entry" (used for both historical dumps and live streaming).

## Implementation Steps

### Phase 1: Remote Node Logic
1. **Update `IceHubLog` & `remote_node`:**
   * Add a `setLogStreamTarget(uint8_t targetId, uint32_t durationMs)` method that returns `true` if a new lease was established.
   * Modify `IceHubLog::addLogEntry` to intercept new lines. If a stream is active, trigger a callback or direct transport call to send the string as `PACKET_MULTIPART` type 2.
2. **Handle Config Request:**
   * In `RemoteNode::handleMessage`, watch for `PACKET_CONFIG` where `command == 2`.
   * When received, call `setLogStreamTarget(0, 15000)`. 
   * If it returns `true` (new lease), iterate through `IceHubLog`'s current buffer and transmit the existing logs to the Hub.

### Phase 2: Hub Caching Logic
1. **Create the Cache:**
   * Add a cache structure to `ApiHandler` or a dedicated `LogManager`: `std::map<uint8_t, std::vector<String>> remoteLogsCache;`
2. **Update `ice_hub.cpp`:**
   * In `handleMessage()`, watch for Multipart messages where `type == 2`.
   * Cast the payload to a String and append it to the `remoteLogsCache` for that specific `sourceId` (keeping the last ~50 lines).

### Phase 3: Web API Updates
1. **Update `api_handler.cpp`:**
   * Track `lastLeaseRenewal[id]`.
   * When `GET /api/nodes/{id}/logs` is hit for a remote node:
     * If `millis() - lastLeaseRenewal[id] > 10000`, send `PACKET_CONFIG (cmd=2)` to renew the lease.
     * Return the current contents of `remoteLogsCache[id]`.

### Phase 4: UI Updates
1. **Update `ui_handler.cpp` (`handleLogsPage`):**
   * Fetch the list of nodes from `/api/nodes` on load and build a `<select>` dropdown.
   * When a new node is selected:
     * Clear the UI log container.
     * Switch the polling interval target to `/api/nodes/{new_id}/logs`.
   * *Optional:* Because it's a live stream, we can just append new lines to the DOM instead of replacing the entire container every 2 seconds.

## Future Considerations
* If multiple users have the web interface open looking at different nodes, the Hub will seamlessly manage multiple leases.
